# LLM Wiki 패턴 (카파시) — 레퍼런스

> 출처: 안드레 카파시 X(트위터) 게시물(2025-04-03) + 국내 해설 영상/리포트.
> 이 문서는 스킬이 "왜 이런 구조인가"를 일관되게 따르도록 하는 배경 설명이다.

## 한 줄 요약
개인이 **목적을 갖고 수집한 지식**을, LLM 에이전트(클로드 코드)가 **소화·연결·정리**해
마크다운 위키로 **복리처럼 축적**하는 프레임워크. 별도 인프라 없이 로컬 폴더 + 마크다운으로 작동.

## 지식 처리 프로세스 (4단계 루프)
1. **수집 (Raw)** — 인터넷에서 목적/의미를 갖고 데이터 수집 → `Raw/` 저장.
2. **인제스트 (LLM 컴파일러)** — 클로드 코드가 Raw 소스를 읽고, 나의 목적·기준·기존 지식과의
   연관성을 확인해 링크 생성·간극 채움·재활용 가능한 형태로 분해.
3. **위키 정리 (Wiki)** — 목차·인덱스가 쌓이며 개념/개체를 분리해 규칙에 따라 정리.
4. **업데이트 루프** — 새 지식이 들어올 때마다 기존과 연결·갱신. 무작위가 아니라 **스키마**(프로세스/원칙/기준) 기반.

## 핵심 개념
- **Gold In, Gold Out**: 목적 있는 수집(=Gold)만이 좋은 산출물을 만든다. GIGO(쓰레기→쓰레기)의 반대.
- **스키마**: LLM에게 매번 랜덤이 아니라 특정 방법을 따르게 지시. 나의 기준·목적을 포함. → 폴더별 `CLAUDE.md`.
- **나의 핵심 맥락(Core Context)**: "나는 누구/왜 기록/어떤 아웃풋" 3문답. AI가 나를 깊이 이해하는 기준.

## LLM Wiki vs RAG
| | LLM Wiki | RAG |
|---|---|---|
| 설정 복잡도 | 낮음 (정리된 마크다운만) | 높음 (임베딩→벡터 DB) |
| 인프라 | 불필요 (로컬 폴더) | 벡터 DB 필요 |
| 검색 신뢰도 | 목록·목차·스키마 기반 → 높을 수 있음 | 쓰레기 데이터 유입 시 저하 |
| 지식 축적 | 복리(연결되어 성장) | 새 문서마다 재업데이트 필요 |

## 한계와 Graphify
- 소스가 많아지면 위키 방식이 비효율적일 수 있음(키워드 검색에 가까워짐).
- **Graphify**: 옵시디언 마크다운을 **지식 그래프**로 만들어 LLM이 연결 기반으로 답하게 함("knowledge graphs for AI coding assistant"). Graph RAG 형태.
- 산출물: `graph.json`(AI 활용), `graph.html`(시각화/클러스터), `graph_report.md`(영문 분석).
- ⚠️ 주의: `pip install graphify`로 설치되는 PyPI 패키지가 영상 속 도구와 동일한지는 미검증. 설치 전 사용자 확인 필요.

## 가장 중요한 것
LLM이 알아서 해주는 게 아니라 **나의 목적**이 핵심이다. "내가 이걸 왜 수집했는지" 답할 수 없으면 제대로 활용하는 게 아니다.
